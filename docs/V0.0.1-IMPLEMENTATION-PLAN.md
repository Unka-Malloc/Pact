# Pact v0.0.1 Implementation Plan

本文是 Pact v0.0.1 的实施计划书，用于把架构讨论拆成可执行、可检查、可验收的开发闭环。

本文不是第六份核心架构文档。本文只定义 v0.0.1 的开发模式、阶段顺序、接口改造、兼容目标、检查点和验收门槛。阶段完成后的长期结论必须回写到：

- `docs/Architecture.md`
- `docs/PROTOCOLS.md`
- `docs/WORKSPACE-ASSET-GOVERNANCE.md`
- `docs/KNOWLEDGE-GOVERNANCE.md`
- `docs/PRODUCTION-CAPABILITY-GAP.md`

Checkpoint、LSM-style ingest、CAS、Merkle DAG、Merkle index、分区事件流和未来分布式无损迁移的算法演进路线单独记录在 `docs/CHECKPOINT-ALGORITHM-EVOLUTION-PLAN.md`。本文的 Phase 0 算法底座检查点必须与该路线保持一致。

## 1. 修正后的开发模式

上一版讨论中有几处过早进入分布式和外部中间件的倾向。v0.0.1 需要修正为“单机端到端优先、通用切面先行、外部能力按顺序闭环”。

| 主题 | 上一版倾向 | v0.0.1 修正 |
| --- | --- | --- |
| 数据库和缓存 | 直接以 PostgreSQL、Redis 作为控制面事实源 | 内部只定义 `StorePort`、`CachePort`、`QueuePort` 等通用切面；v0.0.1 以本地配置文件、现有本地存储和轻量 runtime 为主；PostgreSQL、Redis 只作为外部兼容目标和后续替换基准 |
| 对象存储 | 直接引入 MinIO / S3 artifact store | v0.0.1 不进入分布式对象存储；使用本机受控 staging 和 artifact store；S3/MinIO 作为 v2.0.0 分布式目标 |
| 开源框架 | 把队列、数据库、对象存储都外部化 | 只有安全密钥、证书和高成本安全能力优先借助成熟开源框架；内部交互能用配置文件和轻量本地服务完成的，不引入重框架 |
| 端到端顺序 | 多能力并行铺开 | 严格按顺序推进：本机文件目录 -> 代码库 -> 知识库 -> 云盘；一个链路验收通过后再进入下一个 |
| 验收定义 | Compose 全量分布式栈 | v0.0.1 端到端是从 MCP Plugin 到真实外部服务或本机目标的完整调用链路，不要求分布式部署 |

### 1.1 v0.0.1 设计原则

| 原则 | 要求 | 验收方式 |
| --- | --- | --- |
| 单机优先 | 所有 v0.0.1 能力必须能在单机节点跑通 | `npm run start:all` 或等价单机启动后完成 E2E |
| 通用切面 | 业务模块不得依赖 PostgreSQL、Redis、MinIO 等具体实现语义 | 静态检查禁止业务层直接引用外部实现 adapter |
| 配置驱动 | 模块、连接器、外部目标、权限默认值优先由配置文件声明 | 修改配置后不改代码即可新增目标或禁用目标 |
| 外部能力顺序闭环 | 本机目录、代码库、知识库、云盘按顺序开发和验收 | 每阶段有独立 verifier 和人工/自动 E2E 结果 |
| 治理先行 | 所有外部可见调用都经过 Operation、Policy、Audit、Checkpoint | E2E 输出必须包含 `operationId`、`auditId`、`eventHash`、`beforeRoot`、`afterRoot`、`contentRefs`、`checkpointRef` 或等价记录 |
| 算法底座先行 | 状态、缓存、上下文、知识块和 artifact 从 v0.0.1 起按内容寻址和 Merkle state root 建模 | cache/restore/sync 只引用可验证 ref，不保存不可追溯的裸对象 |
| 上传留档优先 | 上传留档、断点续传、崩溃恢复、审计和 StateCommit 是 v0.0.1 P0；下载/回传链路的 Merkle-Radix 缓存命中率优化后置 | 未通过 LSM ingest、CAS durable archive 和 StateCommit 验收前，不把 prefix cache 作为主目标 |
| 真实存储语义 | 文件、数据库、缓存、云盘和外部服务链路都必须处理真实数据和真实流量；链接、队列、projection、mock、cache hit、provider ref 不能被标记为已保存或已备份 | API、控制台和日志不得把 `queued`、`staged`、`contractVerified`、`projected`、`cached` 说成 `archived`、`committed`、`synced` |
| 预览先行 | 凡是内容到达 Pact 服务器并完成最小留档，就进入 `preview`；权限、风险和审核通过后才进入 `published` | 不把刚到服务器、尚未确认权限和风险的资产展示为 published |
| 轻量框架 | 非安全、非协议、非复杂外部系统能力不引入重型中间件 | 新依赖必须说明替代方案和引入原因 |

### 1.2 v0.0.1 与 v2.0.0 边界

| 能力 | v0.0.1 | v2.0.0 或以后 |
| --- | --- | --- |
| 节点形态 | 单机节点 | 分布式控制面 / 数据面 |
| 数据库 | 通用 `MetadataStorePort`，本地实现优先 | PostgreSQL adapter 作为生产实现 |
| 缓存 | 通用 `CachePort`，内存 / 文件实现优先 | Redis adapter |
| 队列 | 通用 `QueuePort`，本地 durable queue 优先 | Redis / NATS / 其它队列 adapter |
| Artifact | 本地受控 staging / artifact store | S3-compatible / MinIO adapter |
| State/cache substrate | 上传留档 LSM-style ingest、本地 CAS、Merkle DAG manifest、分区 hash chain、Merkle index 接口合同和最小 StateCommit | MPT/Prolly Tree 工业级 adapter、下载/回传高命中率 Merkle-Radix 优化、跨节点同步、远端块存储 |
| Secret | Secret ref 语义和安全框架 adapter | 多后端 secret vault、证书轮换、动态 secret |
| Worker | 逻辑隔离和本地 worker/service runner | 服务级 Worker 集群和 mTLS fleet |

## 3. Phase 0: 通用切面与开发基线

目标：先建立所有后续阶段必须复用的通用切面，避免能力实现时直接写死数据库、缓存、队列、对象存储、云盘或代码库 provider。

| 组件 | 兼容目标 | 开发目标 | 不做 |
| --- | --- | --- | --- |

| ArtifactStorePort | 本机 staging、CAS、未来 S3/MinIO | 统一 artifact ref、读写、清理、hash | 不引入 MinIO 作为 v0.0.1 依赖 |

| Checkpoint/Audit | 现有 Operation Ledger、Checkpoint Tree、本地分区事件流 | 读写都生成可审计记录；event stream 记录 state root 变化，Checkpoint Tree 由投影异步生成 | 不做全局单日志瓶颈，也不把 Checkpoint Tree 当高频写事实源 |

### 3.4 Merkle State Substrate 最小合同

v0.0.1 必须先确定工业级算法底座的接口和引用模型，避免后续从普通状态树迁移到内容寻址结构时重写所有业务链路。第一版实现可以是单机本地文件，但对外合同必须按 Merkle/CAS 建模。

本合同的实现顺序是硬约束：上传留档 LSM-style ingest、CAS durable archive 和 StateCommit 先完成；下载/回传链路的 Merkle-Radix 高缓存命中率、上下文前缀复用和 prefix proof 后置。

| 对象 | 最小字段 | 用途 | 验收 |
| --- | --- | --- | --- |
| `Block` | `cid`、`codec`、`payloadHash`、`byteLength`、`refs`、`createdAt` | 不可变内容块，承载上下文、知识 chunk、artifact、cache value | 相同 canonical payload 得到相同 CID；不同内容不能覆盖旧块 |
| `Manifest` | `rootCid`、`kind`、`refs`、`metadataHash` | Merkle DAG 根对象，描述文件集、context bundle、知识 evidence bundle | 可从 root 遍历全部 refs，并验证缺块 |
| `IndexRoot` | `indexRootCid`、`indexKind`、`namespace`、`keyRange` | Radix/Patricia/Prolly family 的可替换索引根 | 业务只依赖 `MerkleIndexPort`，不写死具体树实现 |
| `IngestSegment` | `segmentId`、`scope`、`level`、`minKey`、`maxKey`、`recordCount`、`rootCid` | LSM-style 上传摄取索引的不可变 segment | 上传 session/chunk metadata 可恢复、可 compact，且不承载文件主体事实 |
| `StateCommit` | `commitId`、`parentCommitIds`、`beforeRoot`、`afterRoot`、`eventHash`、`contentRefs` | Operation 引起的状态变化证明 | 每次写入、授权、恢复、同步都能产生可追溯 state commit |
| `CacheRef` | `cacheKey`、`valueRoot`、`policyHash`、`expiresAt`、`sourceEventHash` | Pact 内部缓存，不依赖 Redis 也能控制一致性和可验证性 | 缓存命中只能返回 root ref 或由 root 验证出来的内容 |

v0.0.1 的算法边界：

- `Checkpoint Tree` 是人可读恢复视图，不是事实源。
- `AuditEventStreamPort` 是操作事实源，分区内用 hash chain 保证追加顺序。
- `ContentAddressedStorePort` 是内容事实源，所有大对象、上下文和知识 evidence 都先进入 CAS。
- `LsmIngestPort` 只负责上传 session、chunk offset、chunk map 和临时 metadata index 的顺序落盘、flush、recover、compaction；文件主体必须进入 CAS/Merkle DAG。
- 下载/回传链路可以先使用简单 deterministic index；Merkle-Radix 的高命中率实现不得阻塞上传留档闭环。
- `MerkleIndexPort` 是索引族接口，第一版可以用简单 sorted chunk/index root 实现，但不得把业务写死成普通 JSON map。
- `StateCommitPort` 把事件流和 Merkle state root 连接起来，所有外部可见 operation 响应都应暴露 state root 或可等价追溯的 ref。

## 4. Phase 1: 本机文件目录闭环

目标：先把最小端到端跑通。Agent 通过 MCP Plugin 访问 Pact，Pact 通过 Sharedspace 管理一个受控本机目录，实现 discover、list、read、write、sync、audit、checkpoint、restore。

### 4.1 阶段总表

| 组件 | 兼容目标 | 开发目标 | 不做 |
| --- | --- | --- | --- |
| Sharedspace | Pact workspace 文件空间、本机目录 | 建立本机目录 adapter 和 workspace projection | 不接云盘 |
| Local Directory Adapter | macOS/Linux 本地目录 | path guard、hash、file metadata、safe write | 不允许任意路径访问；不接受 symlink 作为资产 |
| MCP Plugin | `pact.sharedspace` | Agent 通过语义入口完成文件目录操作 | 不暴露内部文件系统工具列表 |
| Policy/Audit | workspace scope、dataClass、operator | 文件读写都走权限和审计 | 不允许直接写文件绕过 Operation |
| Checkpoint | 文件变更、读取、恢复 | 每次写入、删除、恢复都有 checkpoint | 不用裸 git reset 作为产品恢复 |

## 5. Phase 2: 代码库闭环

目标：在本机目录闭环稳定后，进入 Codespace。第一批代码库兼容目标为 GitHub 和 Gerrit。Codespace 是父能力，内部拆 `RepositoryPort` 和 `ReviewPort`。

### 5.1 阶段总表

| 组件 | 兼容目标 | 开发目标 | 不做 |
| --- | --- | --- | --- |
| Codespace | GitHub、Gerrit | 统一 repo/read/change/review operation | 不把 GitHub/Gerrit 作为两个顶层能力 |

| 偏差 | 风险 | 禁止方式 |

## 11. 待确认设计细节

本节记录后续需要继续拍板的设计问题。优先级定义沿用 `docs/IMPLEMENTATION-DECISION-REGISTER.md`：P0 是不拍板就不能开始正确实现，或会影响权限、安全、协议、数据模型和主链路闭环的问题。

### 已确认设计决议

| 编号 | 决议 | 影响 |
| --- | --- | --- |
| `V001-P0-01` | v0.0.1 repo 内只放模板、默认 manifest 和示例配置；真实运行配置全部放在 `ServerConfig.getDataDir()` 对应的数据目录下，由控制台和 CLI 修改。 | 保持 repo source 与 runtime data 边界清晰；Config Registry、启动脚本、迁移和 doctor 都必须遵守该边界。 |
| `V001-P0-02` | v0.0.1 MCP outlet 硬切为 `pact.discovery`、`pact.knowledge`、`pact.sharedspace`、`pact.codespace`、`pact.skillHub`；不保留旧入口 alias；新版 connector 必须强制重装，doctor 明确提示旧配置不可用并给出重装命令。 | MCP adapter、connector、doctor、Tool Management、协议文档和验证脚本全部按五类新入口实现。 |
| `V001-P0-03` | Intent Operation envelope 固定请求字段：`apiVersion`、`operation`、`subject`、`operatorId`、`agentProfileId`、`workspaceId`、`traceId`、`idempotencyKey`、`intent`、`input`、`dryRun`、`requestedScopes`；固定响应字段：`accepted`、`operationId`、`auditId`、`eventHash`、`beforeRoot`、`afterRoot`、`contentRefs`、`checkpointRef`、`policyDecision`、`resultRef`、`nextRequiredAction`、`warnings`。 | MCP、Console、CLI、Operation Gateway、Audit、Checkpoint、Merkle State 和恢复模型共用同一请求/响应外壳。 |
| `V001-P0-12` | MCP capability discovery 必须按当前 grant 过滤；智能体能看到权限范围内所有可调用 operation，不能看到缺少 scope/toolset/risk 或被 deny 的 operation。 | `pact.capabilities.list`、Tool Management catalog、MCP doctor 和 agent tool planning 都以授权后的真实可见能力为准。 |
| `V001-P0-13` | `/api/mcp/local-grant` 必须有目标匹配机制：没有匹配目标时默认只读；匹配到受支持本机智能体目标后自动授予预定义 safe-write agent toolset，并记录匹配结果和 agentProfileId。 | 支持 Codex、Gemini CLI、Kilo Code、Copilot、OpenClaw、Hermes、Antigravity、OpenCode 等本机 agent 的免手工授权安装，同时保留未知目标只读默认。 |
| `V001-P0-14` | MCP Event Hub 全面实现：`GET /mcp` SSE 除 `tools/list_changed` 外，还必须在 operation 完成或失败后向同一 grant 主动推送 `notifications/pact/operation_reply`。 | 外部智能体不需要轮询即可知道上传、代码评审、同步等长链路最终结果；验证脚本必须覆盖至少一个回信事件。 |
| `V001-P0-15` | 所有写入和上传 operation 必须返回目标回执；智能体必须知道数据送往 Sharedspace、Knowledge、Codespace、Gerrit/GitHub 或其它 provider 的哪个 durable target。 | Codespace 上传必须包含 provider、repository/change/branch/reviewUrl 或等价 durable id；Sharedspace 上传必须包含 workspace/file/artifact 目标。 |
| `V001-P0-04` | v0.0.1 立即引入工业级 Merkle State Substrate 作为算法底座，但不引入外部组件：`LsmIngestPort` 负责上传 session、chunk offset、chunk map 和临时 metadata index 的顺序落盘、flush、recover、compaction；本地 CAS 保存不可变 `Block`；Merkle DAG 表达文件集、context bundle、knowledge evidence 和 artifact；Merkle index family 通过 `MerkleIndexPort` 抽象 Radix/Patricia/Prolly 类索引；`StateCommitPort` 记录 `beforeRoot`、`afterRoot`、`eventHash`、`contentRefs`；审计事实源仍采用本地分区事件流，分区内用 offset/hash chain 保序，Checkpoint Tree 只是从 event stream 和 state root 异步生成的人可读恢复视图。优先级固定为上传留档 LSM-style ingest + CAS durable archive + StateCommit 高于下载/回传链路的 Merkle-Radix cache-hit optimization。 | 避免未来从普通 JSON 状态迁移到 LSM/Merkle/CAS 时重写 operation、upload、cache、restore、sync、context 和 knowledge 链路，同时保留 v0.0.1 单机轻量实现。 |
| `V001-P0-05` | `SecretStorePort` 必须存在，Infisical 是真实 secret 后端的必需依赖，不作为可选增强。测试、mock 和 contract-mode 可以使用 fake SecretStorePort，但真实外部服务接入不得自行实现轻量 secret vault。 | GitHub、Gerrit、Dify、RAGFlow、云盘和后续安全凭据统一走 Infisical-backed secret ref，降低自研密钥管理风险。 |
| `V001-P0-06` | Pact canonical state 是权威，受控本机目录只是 import source 或 projection/mount。只有真实文件字节已经进入 Pact CAS/Merkle manifest 并产生 StateCommit 后，才能对外称为 `archived` 或 `stored`；仅扫描到本机路径、生成同步计划、进入队列或创建 projection 都不能称为已备份。 | 避免用户误以为本机目录里的路径、链接或待处理任务已经被 Pact 保存；Phase 1 的 list/read/write/sync/restore 都必须区分 source、staged、archived 和 projected 状态。 |
| `V001-P0-07` | `dataClass` 是所有资产和外部对象入库的强制字段。MCP、Console、CLI 和内部 adapter 入口都必须在缺失时补默认值 `pending_classification`；该默认值只允许 safe discovery 和元数据级操作，不允许直接读正文、导出、进入上下文或同步到外部目标。 | 字段永远存在，开发复杂度低；同时避免把未分类数据误判成低风险。 |

| `V001-P0-09` | Pact 增加“真实存储语义”硬约束：所有对外状态必须区分 `queued`、`staged`、`archived`、`committed`、`synced`、`projected`、`cached`、`contractVerified`。只有 Pact 本地 CAS/metadata commit 成功或外部服务返回可验证持久化确认后，才能使用 `archived`、`committed`、`synced`。 | 全系统避免“用户以为存了但实际上只存了引用/计划/缓存/模拟结果”的误解行为。 |

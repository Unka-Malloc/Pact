# SplitAll 云端边界

这份文档只回答一个问题：

**为了满足 `Portable + 50MB 内 + Windows + 麒麟 V10`，哪些能力必须上云端，哪些能力可以留在本地。**

## 结论

### 必须上云端

- **视觉 OCR**
  - 图片正文识别
  - 扫描版 PDF 正文识别
  - 图片版 Word / PPT / Excel 中的正文识别
  - 截图、拍照件、扫描件中的文字恢复
- **图文理解**
  - 图表含义提取
  - 截图界面语义理解
  - 流程图、示意图、表格截图的语义归纳
- **知识单元生成**
  - 把原始材料整理成可复用的最小知识文档
  - 标题、标签、去噪、补充结构化字段
- **模拟问答生成**
  - 从知识单元生成问答
  - 避免跨无关 chunk 混拼
- **弱结构材料的语义整理**
  - 扫描件
  - 截图堆
  - 图片与正文交错的复合材料

### 优先本地

- 文件选择、拖拽、目录递归
- 基础 MIME / 扩展名识别
- 文本类文件直接读取
- 已有文本层的 `pdf/docx/office` 提纯
- 确定性规则切分
- 异步任务管理
- 导出 `json/md/docx`
- 配置保存、任务轮询、结果下载

### 本地可选补充

- `Tika` 这类文本提取
- `pdf-parse / mammoth` 这类轻量解析
- `PaddleOCR` 这类本地 OCR sidecar

这些能力可以补，但**不应该成为系统成败的前提**。

## 为什么 OCR 必须以上云端为准

原因不是“本地做不到”，而是**产品约束不允许本地重装运行时**：

- 便携客户端要压在 `50MB` 内
- 目标系统是 `Windows + 麒麟 V10`
- 终端机器性能差
- 不能假设用户机器预装 Python / Java / Office / WebView / OCR runtime

因此：

- 本地 OCR 只能是“有就用，没有也不能影响主流程”
- 真正可靠的 OCR 和图文理解，必须交给**支持视觉输入的云端智能体**

## 推荐架构

### 客户端职责

- 选材
- 上传
- 展示任务状态
- 下载结果
- 做尽可能便宜的本地预处理

### 云端职责

- OCR
- 图文理解
- 语义归纳
- 知识单元生成
- 问答生成
- 最终结果合成

## 对当前项目的落地要求

### 云端模型必须具备

- **视觉输入能力**
  - 至少能稳定处理图片
  - 最好能处理多页图像序列
- **长文本整理能力**
  - 能基于规则切分后的 chunk 再做知识单元生成
- **结构化输出能力**
  - 严格输出约定 JSON

### 本地后端应该继续保留

- 文本层提取
- 规则切分
- 可选 OCR sidecar 接口

但这些都属于：

- 加速项
- 降本项
- 兜底项

而不是核心真相来源。

## 当前决策

- **OCR 以云端智能体视觉能力为主**
- **本地 OCR 仅作为可选补充**
- **不能把本地 OCR 运行时打进 50MB portable 客户端**
